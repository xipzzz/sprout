Stroke by render-size bucket — not one global value

Some stroke variance is legitimate optical sizing, not drift: a 16px inline glyph needs a thinner absolute stroke than a 32px node icon to read at the same weight (how Lucide / SF Symbols stay even). So we lock one stroke per render bucket — then only true outliers need fixing.

16px · inline
1.5px stroke — chevrons in rows, small chips, inline-with-text glyphs
24px · default
2px stroke — the base bucket: nav, gear, bell, mic, volume, users, target, book
32px+ · feature
2.5px stroke — large node / exercise-type icons that anchor a card
Emphasis marks
Check ✓ / X ✗ ride heavier (3–3.5px) on purpose — they sit inside filled circles as bold feedback, not line icons
Inner detail
Decorative sub-strokes (leaf vein, droplet shine, gem facets) stay thin (1–1.6px) — intentionally below the outline weight
Contact sheet — all glyphs at one size (drift jumps out)

Every glyph rendered at the same 26px on one strip. When sizes are equalized, true weight-drift is visible by eye; optically-scaled glyphs that look right at their native size can read a touch heavy/light here — that's expected. Use this as the regression check when adding a glyph.

Rules
One family
24×24 grid, ~2px stroke, round caps/joins, currentColor so icons inherit text color.
Currencies = custom
Streak / gems / water / XP are custom SVG, never emoji — emoji drift across iOS versions.
Feedback = custom
✓ / ✗ are drawn icons (color-blind-safe pairing), not emoji.
Emoji where playful
Fine for in-copy warmth, celebrations and seasonal/content flavor.
Optical sizing
Match visual weight at a given size, not just the box — so a row of icons reads even.
